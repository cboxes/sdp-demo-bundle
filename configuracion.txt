Paso 4. Crear el archivo databricks.yml / Este archivo define el bundle, 
        los archivos de recursos que se incluirán y los ambientes de despliegue.
bundle:
  name: sdp-demo

include:
  - resources/*.yml

workspace:
  host: https://dbc-02ab7e96-d4e0.cloud.databricks.com

targets:

  dev:

    default: true

    workspace:
      root_path: /Workspace/Users/${workspace.current_user.userName}/bundles/${bundle.name}

  prod:

    workspace:
      root_path: /Shared/bundles/${bundle.name}


Paso 5. Crear el Job / Este YAML crea un Workflow con las dependencias entre notebooks.

Archivo:
resources/jobs.yml

resources:

  jobs:

    sdp_pipeline:

      name: SDP Demo Pipeline

      tasks:

        - task_key: setup

          notebook_task:
            notebook_path: ../notebooks/00_setup_lakehouse.py

        - task_key: bronze

          depends_on:
            - task_key: setup

          notebook_task:
            notebook_path: ../notebooks/01_bronze_ingestion.py

        - task_key: silver

          depends_on:
            - task_key: bronze

          notebook_task:
            notebook_path: ../notebooks/02_silver_transformations.py

        - task_key: gold

          depends_on:
            - task_key: silver

          notebook_task:
            notebook_path: ../notebooks/03_gold_models.py

        - task_key: semantic

          depends_on:
            - task_key: gold

          notebook_task:
            notebook_path: ../notebooks/04_semantic_layer.py

***************************************
Cómo lo presentaría en una demostración
***************************************
Podrías explicar el flujo así:

Desarrollo: Los notebooks se desarrollan en Visual Studio Code y se almacenan en Git.
Empaquetado: El proyecto se define mediante un databricks.yml y archivos YAML de recursos.
Validación: databricks bundle validate comprueba que la configuración sea correcta.
Despliegue: databricks bundle deploy crea o actualiza notebooks y workflows en el workspace.
Ejecución: databricks bundle run sdp_pipeline lanza el pipeline completo.
Resultado: Los datos quedan organizados en sdp_catalog (Bronze, Silver y Gold), mientras que los modelos analíticos pueden 
           consumir también los esquemas main.analytics y main.default.

Esta demostración evidencia buenas prácticas de Infrastructure as Code, CI/CD, versionamiento con Git y automatización de 
pipelines, sin necesidad de crear o modificar recursos manualmente desde la interfaz de Databricks.
